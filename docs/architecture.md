# Runtime Architecture ‚Äî fourth-path-gemma

## Pipeline flow

The default entry point is `run_pipeline()` in `main.py`. The execution path is:

```
classify_input(prompt)
  -> input_canon_issue : refuse immediately (REFUSAL_OUTPUT, no generation)
  -> forward_to_main_agent :
       for attempt in 1..MAX_ATTEMPTS (3):
         generate_candidate(Main Agent)
           -> classify_candidate(candidate)
                -> local_issue      : retry (or refuse on last attempt)
                -> canon_issue      : retry with canon-clause signal (or refuse on last attempt)
                -> forward_to_cold_eyes :
                     mechanical_cold_eyes_review(policy, candidate)
                       -> pass : return candidate  [pipeline exits]
                       -> fail : retry with canon-clause signal (or refuse on last attempt)

Unified Refusal: REFUSAL_OUTPUT  (fixed string, issued outside the generation loop)
```

`REFUSAL_OUTPUT` is a module-level constant. It is never generated by a model call; it is set as the default `final_output` before the loop begins and is returned whenever the loop exits without a `pass` verdict.

## Role of each layer

### Main Agent

The Main Agent is a reasoning-only layer. Its system prompt (`MAIN_AGENT_SYSTEM_PROMPT`) explicitly instructs the model to produce a candidate answer and nothing else: it holds no prohibitions, performs no safety adjudication, writes no refusal messages, and cannot approve its own output. The Main Agent never sees canon.md or Cold Eyes feedback directly; when retrying, it receives only a brief signal ("a previous candidate violated clause X") appended to the original user prompt.

### classify\_candidate / classify\_input

Both classifiers are fast, deterministic regex prefilters implemented entirely in Python (no model calls). `classify_input` checks the raw user prompt before any generation occurs; `classify_candidate` checks each generated candidate before it reaches Cold Eyes. They route traffic ‚Äî flagging obvious canon violations and local formatting issues ‚Äî but they are not final adjudicators. A candidate that clears both classifiers still must pass Mechanical Cold Eyes before it is returned to the caller.

### Mechanical Cold Eyes (`audit/` package)

Mechanical Cold Eyes is the deterministic final adjudicator. It is implemented in `audit/engine.py`, `audit/policy.py`, and `audit/matcher.py` with no LLM dependency. At runtime, `mechanical_cold_eyes_review()` in `main.py` calls `run_audit(candidate, policy_path)`, which loads a JSON policy file, evaluates each rule's regex patterns against the candidate text, and returns a structured `AuditResult`. The verdict is binary: `pass` (no matches) or `fail` (one or more matches). The Cold Eyes layer is fail-closed: any error in policy loading, regex compilation, or matching produces a `fail` verdict, never a `pass` (see [Fail-closed properties](#fail-closed-properties) below).

### semantic\_audit.py / cold\_eyes\_review()

`cold_eyes_review()` is a legacy LLM-based reviewer. It sends the canon and the candidate to a second model invocation using `COLD_EYES_SYSTEM_PROMPT` and parses the JSON response. This function is **not called in the default pipeline**; `mechanical_cold_eyes_review()` is used instead. `cold_eyes_review()` is exposed for experimental and diagnostic use only and cannot issue a final `pass` verdict in normal operation.

## Fail-closed properties of Mechanical Cold Eyes

`run_audit()` returns `verdict="fail"` unconditionally in each of the following conditions, regardless of the candidate content:

| Condition | Triggered by |
|---|---|
| Policy file unreadable or not valid UTF-8 | `load_policy()` raises `PolicyError` with reason `cannot read policy file` |
| Policy file is not valid JSON | `load_policy()` raises `PolicyError` with reason `invalid policy JSON` |
| Policy structure invalid (missing fields, wrong types) | `load_policy()` raises `PolicyError` (e.g., missing `version`, missing `rules`, rule missing `id`) |
| Unknown severity value in a rule | `load_policy()` raises `PolicyError` with reason `unknown severity` (valid values: `critical`, `major`, `minor`) |
| Invalid regex in a pattern | `compile_pattern()` raises `MatcherError`, caught in `run_audit()` as `matcher_error` |

In all error cases the returned `AuditResult` has `verdict="fail"`, `canon_clause=None`, and a `reason` string prefixed with the error type. No exception propagates to the caller.

## Logging privacy stance

Each pipeline run writes a JSONL file under `runs/` via `write_audit_log()`. The persisted log records routing and verdict metadata only.

**Recorded per attempt** (`AuditEntry.log_dict()`):
- `run_id` ‚Äî timestamp + 8-hex UUID run identifier
- `attempt` ‚Äî attempt number (1‚Äì3)
- `classify_route` ‚Äî routing decision (`forward_to_cold_eyes`, `local_issue`, `canon_issue`, `input_canon_issue`)
- `cold_eyes_verdict` ‚Äî `pass` / `fail` / `null`
- `canon_clause` ‚Äî `C1` / `C2` / `C3` / `null`
- `final_status` ‚Äî `pass` / `refused` / `null` (only on the last attempt entry)

**Recorded for the final event**:
- `run_id` ‚Äî same run identifier used by attempt entries
- `status` ‚Äî final pipeline status
- `attempts` ‚Äî final attempt count
- `output_chars` ‚Äî character count of the final output

**Not recorded in the persisted JSONL log**:
- The user prompt text
- Any candidate text generated by the Main Agent
- The full final output (only its character length is logged)
- Any user identity

`AuditEntry.public_dict()` follows the same omissions as `AuditEntry.log_dict()`, except that it omits `run_id` for user-facing display. `RunResult.public_dict()` is different: it is used for JSON output mode (`--json`) and includes the full `output` field returned to the caller. Therefore, the JSON command output is not a privacy-equivalent audit log; the privacy claim above applies to persisted JSONL logs under `runs/`.
